Modelagem do Sistema

6 entidades, SQL completo e tabela de permissões por role.

6 Entidades do MVP

profiles  —  sectors  —  properties  —  visits  —  evidences  —  activity_logs

-- ========================================
-- ZOONOSEAPP MVP — SCHEMA COMPLETO
-- ========================================

-- 1. SETORES (bairros / regiões de cobertura)
CREATE TABLE sectors (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        TEXT NOT NULL,
  municipality TEXT NOT NULL DEFAULT 'Uberlândia',
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 2. PERFIS DE USUÁRIO (extende auth.users)
CREATE TABLE profiles (
  id         UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  name       TEXT NOT NULL,
  role       TEXT NOT NULL CHECK (role IN ('agent','coordinator','manager','admin')),
  sector_id  UUID REFERENCES sectors(id),
  active     BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 3. IMÓVEIS
CREATE TABLE properties (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code       TEXT UNIQUE NOT NULL,   -- ex: UBL-0042
  address    TEXT NOT NULL,
  sector_id  UUID REFERENCES sectors(id),
  type       TEXT DEFAULT 'residencial'
               CHECK (type IN ('residencial','comercial','terreno','especial')),
  lat        DOUBLE PRECISION,
  lng        DOUBLE PRECISION,
  active     BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 4. VISITAS
CREATE TABLE visits (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  property_id UUID NOT NULL REFERENCES properties(id),
  agent_id    UUID NOT NULL REFERENCES profiles(id),
  status      TEXT NOT NULL CHECK (status IN (
                'visitado_sem_foco',
                'visitado_com_achado',
                'fechado',
                'recusado',
                'nao_localizado',
                'pendente_revisao'
              )),
  notes       TEXT,
  visited_at  TIMESTAMPTZ NOT NULL,
  synced_at   TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 5. EVIDÊNCIAS (fotos por visita)
CREATE TABLE evidences (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  visit_id     UUID NOT NULL REFERENCES visits(id) ON DELETE CASCADE,
  storage_path TEXT NOT NULL,  -- caminho no Supabase Storage
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- 6. LOGS DE ATIVIDADE
CREATE TABLE activity_logs (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID REFERENCES profiles(id),
  action    TEXT NOT NULL,   -- ex: 'create_visit', 'export_csv'
  entity    TEXT NOT NULL,   -- ex: 'visits', 'properties'
  entity_id UUID,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- ========================================
-- RLS POLICIES (Row Level Security)
-- ========================================

ALTER TABLE visits ENABLE ROW LEVEL SECURITY;

-- Agentes vêem apenas suas próprias visitas
CREATE POLICY "agents_own_visits" ON visits
  FOR ALL USING (
    agent_id = auth.uid()
    OR EXISTS (
      SELECT 1 FROM profiles p
      WHERE p.id = auth.uid()
        AND p.role IN ('coordinator','manager','admin')
    )
  );

-- Agentes criam apenas visitas com seu próprio ID
CREATE POLICY "agents_insert_own" ON visits
  FOR INSERT WITH CHECK (agent_id = auth.uid());

-- INDEX para performance
CREATE INDEX idx_visits_property  ON visits(property_id);
CREATE INDEX idx_visits_agent     ON visits(agent_id);
CREATE INDEX idx_visits_status    ON visits(status);
CREATE INDEX idx_properties_code  ON properties(code);
CREATE INDEX idx_properties_sector ON properties(sector_id);

05

Fluxos e Telas

App do agente (Android) e painel do coordenador (Web). Cada tela com suas ações.

📱 App do Agente — 8 Telas
1

Login

Campo e-mail + senha. Botão "Entrar". Erro inline se credenciais inválidas. Token JWT salvo no MMKV. Redireciona para Home.

2

Home / Busca de Imóvel

Campo de busca por código ou endereço. Lista de imóveis do setor do agente. Badge de status da última visita. Busca funciona offline (cache MMKV).

3

Detalhe do Imóvel

Código, endereço, tipo. Últimas 5 visitas com status e data. Botão "Registrar Visita" em destaque.

4

Registrar Visita

6 botões de status (cores diferentes). Campo de nota opcional. Botão de câmera para foto. Confirmação antes de salvar.

5

Confirmação de Visita

Resumo: imóvel, status selecionado, nota, foto (thumbnail). Botões "Confirmar" e "Editar". Salva localmente, enfileira sync.

6

Histórico Pessoal

Lista das visitas do agente (hoje, últimos 7 dias). Filtro por status. Contador de visitas realizadas.

7

Status de Sync

Número de visitas pendentes de envio. Botão "Sincronizar agora". Indicador visual: verde (sincronizado) / amarelo (pendente) / vermelho (erro).

8

Configurações

Nome do agente, setor, versão do app. Botão "Sair". Limpar cache local.

💻 Painel do Coordenador — 5 Telas
1

Dashboard Principal

Métricas: total imóveis, % visitados hoje, achados, recusas, pendências. Tabela de agentes ativos com contagem.

2

Lista de Imóveis

Tabela filtrável por setor e status. Coluna com última visita e agente. Botão exportar CSV.

3

Detalhe do Imóvel (Web)

Histórico completo de visitas com fotos. Timeline visual por data.

4

Lista de Agentes

Agentes por setor. Visitas realizadas hoje e no período. Status online/offline.

5

Exportação

Filtros: período, setor, status. Download CSV. Nome do arquivo com data.

⚙ Admin — 3 Telas
A

Criar Usuário

Formulário: nome, e-mail, senha temporária, role, setor. Confirmação por e-mail automática via Supabase Auth.

B

Gerenciar Setores

Criar e editar setores. Associar imóveis a setor por CSV.

C

Import CSV Imóveis

Upload de planilha. Preview das primeiras 5 linhas. Confirmar import. Log de erros por linha.

🕐 Fluxo Crítico do MVP (caminho feliz)

Login → Buscar imóvel por código → Ver detalhe + histórico → Registrar visita → Selecionar 1 dos 6 status → Tirar foto → Adicionar nota → Confirmar → Salvar localmente → App sincroniza automaticamente ao conectar → Coordenador vê no painel.

O que Vale Terceirizar

O que contratar, por que, qual perfil buscar e como estruturar milestones.

💰 Budget e Perfil

Budget: R$ 8.000–15.000 por projeto (4–5 semanas). Perfil ideal: freelancer React Native + Supabase, portfólio B2B publicado no GitHub, disponível por projeto (não CLT). Aceita pagar por milestone, não por hora.

O que Terceirizar Por que Não Fazer Sozinho Perfil Necessário Formato Risco se Founders Tentarem
Setup Supabase + modelagem (T01) RLS policies erradas criam brechas de segurança silenciosas Full-stack com Supabase ou PostgreSQL Task única, 4h Alto — dados vazam entre usuários
App agente telas core (T03–T07) React Native tem quirks de navegação e estado que são lentos de aprender React Native 2+ anos de experiência Milestone M1 Crítico — pode levar 10x mais tempo
Offline + sync (T09) Conflict resolution, filas, NetInfo, retry logic — não é trivial Experiência específica com MMKV ou WatermelonDB Task dedicada, 8h Crítico — app não funciona no campo sem isso
Painel coordenador (T10–T14) Next.js + Supabase Realtime + queries agregadas requerem experiência Full-stack Next.js + SQL Milestone M2 Médio — painel funciona mas sem qualidade
Auth + RLS completo JWT, cookies, session management, policies por role — área de segurança Full-stack com Supabase Auth Incluso no M1 Alto — brechas de acesso entre roles
Upload de foto + Storage (T08) Compresso, permissões da câmera Android, upload com retry React Native com expo-image-picker Incluso no M1 Médio — fotos não aparecem ou sem compressaão
Build APK EAS (T17) Configuração de certificate, signing, eas.json — 1h se sabe, 4h se não sabe Qualquer dev Expo experiente Task final, 2h Baixo — só perde tempo
Como Contratar Certo

1. Exigir portfólio publicado com pelo menos 1 app React Native no GitHub ou Play Store.
2. Teste técnico de 2 horas: "Crie uma tela de login com Expo + Supabase Auth que salva o token no MMKV."
3. Pagar por milestone, nunca por hora:
  • M1 (R$ 5.000): Scaffold + auth + 8 telas app agente com offline
  • M2 (R$ 4.000): Painel coordenador + export CSV + deploy Vercel
  • M3 (R$ 1.000–2.000): APK EAS + QA + correções
4. Check diário de 30 min via WhatsApp. Sem reuniõo, só status + bloqueios.

O que NãO Fazer Agora

15 erros comuns de founders técnicos que atrasam o MVP sem adicionar valor real.

1. Offline perfeito antes de validar adoção

Conflict resolution distribuído é engenharia de nível Notion/Linear. Para MVP, sync manual por botão é suficiente e demonstrável. Offline perfeito só depois que a prefeitura assinar.

2. Mapas e heatmaps

Bonito na apresentação, inútil sem dados reais de GPS coletados em campo. Adiciona 2 semanas de desenvolvimento e não muda a decisão da prefeitura no piloto.

3. App iOS agora

Prefeituras brasileiras usam Android. iOS exige Mac + Apple Developer Account ($99/ano) + processo de review. Adicione depois de ter contrato.

4. Play Store antes do piloto

APK via link direto é mais rápido, flexível e suficiente para piloto. Play Store tem review de 3–7 dias e requer conta de desenvolvedor. Publique só na versão comercial.

5. Notificações push

Requer Firebase Cloud Messaging, permissões do usuário, lógica de gatilho. Para o piloto, o coordenador olha o painel. Push notification é conforto, não necessidade do MVP.

6. Multi-tenant desde o início

Adicionar organização como camada de isolamento triplica a complexidade do schema e da RLS. Um cliente primeiro, depois multi-tenant. Não tem como vender para 2 prefeituras ao mesmo tempo no piloto.

7. Dashboard "bonito" com gráficos

Recharts, Chart.js, D3 — adicionam complexidade sem aumentar a decisão da prefeitura. Números grandes em cards convencem mais que gráficos de linha em MVP. Gráficos depois de ter dados reais de 90 dias.

8. Contratar designer antes de validar

Design system, biblioteca de componentes, Storybook — custo alto antes de saber se o produto tem adoção. Use shadcn/ui ou Tailwind direto. Design profissional na versão paga.

9. Firebase em vez de Supabase

Firebase Firestore não tem SQL real, o que torna queries agregadas (métricas do dashboard) muito mais complexas. Supabase é PostgreSQL — qualquer dev conhece SQL. Não mude a stack por preferência.

10. Backend customizado Node/Express

Supabase Edge Functions e RLS eliminam a necessidade de backend customizado para o MVP. Adicionar Express/NestJS é mais uma coisa para manter, testar e deployar. YAGNI.

11. Montar time antes de ter cliente

Designer + product manager + segundo dev antes do primeiro contrato é burn de caixa sem retorno. Um dev freelancer por projeto é o modelo certo até ter receita recorrente.

12. Billing no sistema

Stripe, assinatura, plãos, invoices dentro do produto — o piloto é gratuito por definição. Invoice manual via PDF ou Conta Azul até ter 3 clientes pagantes. Não construa billing antes de ter alguém para cobrar.

13. Reuniões longas sem código na tela

Qualquer reunião de mais de 30 minutos sobre o produto que não termina com uma decisão ou uma tarefa é tempo perdido. Check diário de 15 min com o dev. Reunião semanal de 30 min de retrospectiva. Nada mais.

14. Relatórios PDF automáticos

Puppeteer, html-pdf, LaTeX — relatório PDF é V2. A prefeitura aceita CSV no Excel para o piloto. PDF bonito impressiona, mas não é critério de aceite do piloto. Adicione quando pedirem.

15. Esperar o produto "estar pronto"

O produto nunca vai estar pronto. Demo com APK funcionando e 50 imóveis seed é suficiente para ter a primeira conversa real. A validação não acontece no Figma — acontece com um agente real em campo.
